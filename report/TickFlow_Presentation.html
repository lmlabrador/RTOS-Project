RTOS Concepts — Scheduling Policy & Justification

Fixed-Priority Preemptive Scheduling

Highest-priority ready job always runs — can preempt anything below it.

π=1 T1  HI runs first
↓ preempted if T1 arrives
π=2 T2  LO
↓
π=3 T3  HI runs last
RTA: Ri = Ci + Σhp(i) ⌈Ri/Tj⌉·Cj

Extended Task Control Block (TCB)

Standard TCB + three new MC fields:

periodTi — activation interval (ms)
deadlineDi — relative deadline (ms)
priorityπi — dispatch order
criticalityLi ∈ {LO, HI}NEW
wcetLoCLOi — optimistic boundNEW
wcetHiCHIi — true worst-caseNEW

// SchedulerEngine.ts — called every tick step(tasks: TaskModel[], tickMs: number) { // 1. advance time, release jobs, check deadlines this.state.time += tickMs; this.releaseJobs(tasks, currentTime); this.detectDeadlineMisses(currentTime); // 2. pick highest-priority ready job const selected = this.selectReadyJob(); // 3. execute for one tick selected.executedTime += tickMs; // 4. ← MODE SWITCH CHECK const exceeded = this.state.mode === "LO" && selected.criticality === "HI" && selected.executedTime > selected.wcetLo; if (exceeded) this.triggerModeSwitch(); } private triggerModeSwitch() { this.state.mode = "HI"; // flip mode // remove ALL LO jobs immediately const suspended = readyJobs.filter( j => j.criticality === "LO" ); // log each as "suspension" event suspended.forEach(j => pushEvent("suspension")); readyJobs = readyJobs.filter(j => j.criticality === "HI"); }

What triggers the switch?

  • Mode = LO
  • Running task = HI-criticality
  • executedTime > wcetLo

What happens immediately?

  • Mode flips to HI
  • All LO jobs removed from ready queue
  • Each logged as suspension event
  • LO jobs never re-released

35% overrun probability — HI job budget sampled above wcetLo. The first overrun triggers the mode switch; the system stays in HI mode for the rest of the run.

Learning Outcomes Reflection
Slides 11–12 · Key Insights

LO1 — Key Takeaway

RMS and DMS are easy to implement. The real insight came from the optimizer — comparing 4 strategies on a live task set showed DMS wins for constrained deadlines in a way no textbook derivation could.

"We understood why DMS > RMS only after seeing the optimizer score them."

LO3 — Key Takeaway

Priority inversion is usually a semaphore problem. In MC scheduling: a LO task in the ready queue blocks a HI task. Mode-switch resolves it — suspend all LO, guarantee CPU to HI.

"Suspending T2 is not a bug — it is the concurrency solution."

LO4 — Key Takeaway

TickFlow's mode-switch in software mirrors what a microkernel enforces in hardware — isolating HI partitions from LO ones. This connection made kernel architecture comparison concrete, not just theoretical.

"Mode-switch ≈ ARINC 653 partition scheduling at the hardware level."
Overall Building a simulator forces precision that reading cannot — every ambiguity in theory becomes a bug in code.
