Problem Statement

The Core Challenge

  • Safety-critical systems must serve tasks of vastly different importance on shared hardware.
  • A collision-avoidance routine and a cabin-temperature display share one CPU — a missed deadline in one is catastrophic, in the other inconvenient.
  • Traditional RTOS schedulers treat all tasks equally — no mechanism to distinguish criticality levels at runtime.

The Simulator Gap

  • Physical RTOS platforms require specialized hardware and lengthy toolchain setup.
  • Existing tools (Cheddar, MAST) are desktop-only, CLI-based, or commercial.
  • Mode switching is difficult to visualize without live, interactive feedback.
  • Need: An accessible, visual, browser-native MC scheduling tool for learning and analysis.

Motivation & Objectives

Industry Motivation

  • ARINC 653 (Avionics) and AUTOSAR (Automotive) mandate mixed-criticality task management.
  • ISO 26262 & DO-178C require formal schedulability evidence for safety certification.
  • SWaP constraints (Size, Weight, Power) force co-hosting tasks of different safety levels.

Project Objectives

  • Implement the Vestal MC model faithfully in a browser simulator.
  • Provide real-time visualization of Gantt charts, events, and mode switches.
  • Implement Response Time Analysis (RTA) for schedulability verification.
  • Develop an optimization engine for priority assignment recommendations.
  • Validate against established scheduling theory.

RTOS Concepts: Mixed-Criticality Scheduling

Vestal Model (2007)

  • Each task has criticality level: LO or HI
  • HI tasks carry two WCET budgets:
       CLO — safe pessimistic estimate
       CHI — true worst-case (CHI ≥ CLO)
  • LO mode: All tasks run; HI tasks use CLO
  • Mode switch: If HI task exceeds CLO, switch to HI mode
  • HI mode: LO tasks are suspended; HI tasks use CHI

Why This Works

🟢 LO Mode  (normal)

Higher utilization. Optimistic assumption that HI tasks behave well.

🔴 HI Mode  (triggered)

LO tasks sacrificed. Full CPU bandwidth reserved for safety-critical HI tasks.

ULO = Σ CLOi / Ti < 1.0
UHI = ΣHI CHIi / Ti < 1.0

RTOS Concepts: Scheduling & Analysis

Fixed-Priority Preemptive (FPPS)

  • Highest-priority ready task always runs
  • Can preempt lower-priority tasks instantly
  • Priority: RMS (1/T) or DMS (1/D)
  • Dominant in safety-critical standards

Response Time Analysis

Ri = Ci + Σj∈hp(i) ⌈Ri/Tj⌉ · Cj
  • Iterative fixed-point computation
  • Task schedulable if Ri ≤ Di
  • Computes worst-case response time

Key Implementation Challenges

Challenge 1 — Mode Switch Timing

Determining exactly when a HI task crosses its CLO budget within a discrete simulation tick.

if executedMs > wcetLo → triggerModeSwitch()

Solution: Check after every tick increment. Tick sizes (1/2/5ms) bound detection latency.

Challenge 2 — Memory Bounds

Long simulations generate thousands of events and Gantt segments.

Solution: Fixed circular buffers — 400 events, 1800 segments, 400 mode switches — keeping heap footprint under 400KB.

Challenge 3 — RTA Convergence

Iterative RTA may diverge for unschedulable task sets.

Max iterations = 100 → mark FAILED

The report modal flags non-converging tasks in red with the last computed Ri value.

Challenge 4 — Browser Timer Jitter

setInterval is not real-time. Callbacks drift under CPU load.

Solution: All scheduling decisions use simulation time, not wall-clock time. Wall-clock only determines UI update rate.

Implementation: Key Solutions

Scheduler Step Function

step(tasks, tickMs) {
  this.time += tickMs;
  this.releaseJobs(tasks);
  this.detectDeadlineMisses();

  const running = this.selectReadyJob(tasks);
  if (running) {
    running.executedMs += tickMs;
    // record Gantt segment
    this.segments.push({ taskId, start, end, mode });

    // Mixed-criticality mode switch check
    if (this.mode === 'LO' && task.criticality === 'HI'
        && running.executedMs > task.wcetLo) {
      this.triggerModeSwitch(tasks); // suspend LO tasks
    }
    if (running.executedMs >= running.budget)
      this.completeJob(running);
  }
}

Execution Budget Sampling

// 35% overrun probability for HI tasks in LO mode
if (mode === 'LO' && Math.random() < 0.35) {
  return wcetLo + 1 + rand(wcetHi - wcetLo);
}
return rand(wcetLo) + 1;

Learning Outcomes Reflection

LO1 — Task Modeling

Implementing the TaskModel interface deepened understanding of periodic task abstraction and the limitations of modeling aperiodic workloads within a fixed-period framework.

LO2 — Scheduling Analysis

Computing RTA by hand for the default task set, then verifying against the simulator, solidified the fixed-point iteration concept and revealed corner cases in convergence behavior.

LO3 — Algorithm Implementation

Writing the scheduler step() function required careful translation of textbook scheduling algorithms into working, testable code with correct preemption semantics.

LO4 — Mixed-Criticality Trade-offs

Observing live mode switches revealed the deliberate design decision behind MC scheduling: accepting LO-task failures is the safety mechanism, not a bug. This mindset shift is the core insight of MC systems.

LO5 — Formal Analysis Tools

The 8-configuration optimization engine showed that formal analysis can guide priority assignment beyond intuition. Conservative Demotion heuristic revealed over-provisioned tasks where HI classification added no benefit.

Key Insight: WCET margin selection is not arbitrary — a 35% overrun rate causes ~7 mode switches per hyperperiod, completely starving LO tasks. Tight margins enable better utilization at the cost of safety.

Challenges & Lessons Learned

Technical Challenges

  • Browser timer jitter — solved by simulation-time abstraction
  • Memory growth — solved with bounded circular buffers
  • RTA divergence — solved with iteration limit + FAILED flag
  • No mode recovery — deliberate simplification (open research problem)

Design Lessons

  • Separation of concerns — engine independence enabled isolated testing
  • TypeScript strict mode — caught interface mismatches early
  • Zustand over Redux — 80% less boilerplate for same functionality

What I Would Do Differently

  • Add mode recovery protocol for complete Vestal model compliance
  • Implement EDF scheduling as comparison against RMS/DMS
  • Add Monte Carlo simulation over N hyperperiods for statistical schedulability
  • Export task sets to FreeRTOS configuration files for hardware validation
  • Support multiprocessor global/partitioned scheduling

Validated Approach: Layered architecture (engine / store / components) made it trivial to swap priority strategies in the optimizer without touching visualization code.

Conclusion & Future Work

Summary

  • Built TickFlow: a browser-based mixed-criticality scheduling simulator implementing the Vestal MC model.
  • Implemented fixed-priority preemptive scheduling with automatic LO→HI mode switching.
  • Verified default task set schedulable in both modes via RTA (5/5 tasks pass).
  • Optimization engine evaluates 8 configurations and recommends optimal priority assignment.
  • System operates at 59.2% / 73.3% utilization in LO/HI modes — well below capacity.

Recommendations

  • Always verify schedulability with formal RTA before deployment
  • Select WCET budgets conservatively — overrun rate directly impacts LO throughput
  • Prefer DMS over RMS for constrained-deadline task sets
  • Maintain strict architectural separation between scheduling logic and platform code

Future Work

  • EDF & sporadic task support
  • Multiprocessor global scheduling

BACKUP — Q&A

Anticipated Q&A

Q: Why not use EDF instead of FPPS?

EDF achieves 100% utilization for implicit-deadline task sets, but: (1) EDF requires dynamic priority recomputation each job release, adding runtime overhead; (2) safety-critical standards (ARINC 653, AUTOSAR) mandate fixed priorities for auditability; (3) EDF's behavior under overload is unpredictable — it may miss any task's deadline, whereas FPPS guarantees lower-priority tasks miss first.

Q: Why no mode recovery (HI→LO)?

Mode recovery is an open research problem. Naively re-admitting LO tasks after a HI mode event can violate HI-task deadlines during the transition window. Formal recovery protocols (e.g., Baruah 2011) require complex cross-mode schedulability analysis. TickFlow omits this to keep the model tractable and pedagogically clear.

Q: How accurate is the WCET sampling model?

The 35% overrun probability is a pedagogical approximation. Real WCET analysis uses static analysis (AbsInt aiT, OTAWA) or measurement-based bounds. The simulator's probabilistic model demonstrates the qualitative effect of WCET margin selection on mode switch frequency without claiming quantitative accuracy for specific hardware.

Q: Does this scale to more tasks?

Yes. The engine is O(n) per tick for n tasks. RTA is O(n²) per analysis. Circular buffers bound memory regardless of task count. Tested with up to 10 tasks without performance degradation. Gantt rendering at 1800 segments targets ~900ms window regardless of task count.
